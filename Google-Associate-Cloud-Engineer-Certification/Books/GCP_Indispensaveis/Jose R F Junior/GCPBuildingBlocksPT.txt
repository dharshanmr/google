Anycast
Anycast é umarede endereçamento e roteamento de metodologia dena qual um único endereço de destino tem vários caminhos de roteamento para dois ou mais destinos de ponto de extremidade. Os roteadores selecionarão o caminho desejado com base no número de saltos, distância, menor custo, medições de latência ou com base na rota menos congestionada. As redes Anycast são amplamente usadas para produtos de rede de entrega de conteúdo(CDN) para aproximar seu conteúdo do usuário final.

O Internet Protocol e outros sistemas de endereçamento de rede reconhecem cinco métodos de endereçamento principais: O
Unicast endereçamentousa uma associação um-para-um entre um remetente e um destino: cada endereço de destino identifica exclusivamente um único terminal do receptor.
Broadcast usa uma associação de um para todos; um único datagrama de um remetente é roteado para todos os possíveis vários pontos de extremidade associados ao endereço de broadcast. A rede replica automaticamente os datagramas conforme necessário para alcançar todos os destinatários dentro do escopo da transmissão, que geralmente é uma sub-rede inteira da rede.
Omulticast endereçamentousa uma associação um-para-muitos-de-muitos ou muitos-para-muitos-de-muitos; os datagramas são roteados simultaneamente em uma única transmissão para muitos destinatários. Difere do broadcast em que o endereço de destino designa um subconjunto, não necessariamente todos, dos nós acessíveis.
O endereçamento Anycast é uma associação de um para muitos, em que os datagramas são roteados para qualquer membro único de um grupo de receptores potenciais, todos identificados pelo mesmo endereço de destino. O algoritmo de roteamento seleciona o único receptor do grupo com base na métrica de roteamento mais barata. Na prática, isso significa que os pacotes são roteados para o membro topologicamente mais próximo de um grupo anycast.
Geocast refere-se à entrega de informações a um grupo de destinos em uma rede identificada por suas localizações geográficas. É uma forma especializada de endereçamento multicast usada por alguns protocolos de roteamento para redes móveis ad hoc.

AS Sistemas Autônomos ASN Número de Sistemas Autônomos
Na Internet, um sistema autônomo (AS) é um conjunto de prefixos de roteamento Internet Protocol (IP) conectados sob o controle de um ou mais operadores de rede em nome de uma única entidade administrativa ou domínio que apresenta um política de roteamento comum e claramente definida para a Internet.


Originalmente, a definição exigia o controle por uma única entidade, normalmente um provedor de serviços da Internet ou uma organização muito grande com conexões independentes para várias redes, que aderem a uma política de roteamento única e claramente definida, como originalmente definido na RFC 1771. [2] A nova definição no RFC 1930 entrou em uso porque várias organizações podem executar o Border Gateway Protocol (BGP) usando números privados de AS para um ISP que conecta todas essas organizações à Internet. Embora possa haver vários sistemas autônomos suportados pelo ISP, a Internet só vê a política de roteamento do ISP. Esse ISP deve ter umoficialmente registrado número de sistema autônomo(ASN).

grupos de instâncias gerenciadas do autoescalador oferecem recursos de escalonamento automático que permitem adicionar ou excluir automaticamente instâncias de um grupo de instâncias gerenciadas com base em aumentos ou diminuições na carga. O escalonamento automático ajuda seus aplicativos a lidar com o aumento do tráfego e reduz o custo quando a necessidade de recursos é menor. Você apenas define a política de escalonamento automático e o autoescalador executa a escala automática com base na carga medida.
O escalonamento automático funciona aumentando ou diminuindo seu grupo de instâncias. Ou seja, ele adiciona mais instâncias ao seu grupo de instâncias quando há mais carga (upscaling) e exclui instâncias quando a necessidade de instâncias é reduzida (downscaling).

Políticas de disponibilidade As
opções de disponibilidade determinam como as instâncias de VM se comportam quando ocorrem eventos de manutenção. Como definir uma instância para migrar ao vivo quando ocorre um evento de manutenção.
Quando há eventos de manutenção, como atualizações de hardware ou software, que exigem que o Google mova sua VM para uma máquina host diferente, o Google Compute Engine gerencia automaticamente o comportamento de agendamento de suas instâncias. O Compute Engine Live migra suas instâncias de VMs se você tiver configurado a política de disponibilidade da instância para usar a migração ao vivo. Isso impede que seus aplicativos sofram interrupções durante esses eventos. Como alternativa, você também pode optar por encerrar suas instâncias durante esses eventos, em vez de migrá-las ao vivo.

Atualização canário
O recurso Instance Group Updater permite executar atualizações canárias, para que você possa testar suas atualizações em um subconjunto aleatório de instâncias antes de confirmar totalmente a atualização.
Uma atualização canary é uma atualização aplicada a um número parcial de instâncias no grupo de instâncias. As atualizações das Canárias permitem que você teste novos recursos ou atualizações em um subconjunto de instâncias, em vez de implantar uma atualização potencialmente destrutiva em todas as suas instâncias. Se uma atualização não estiver indo bem, você só precisará reverter um pequeno número de instâncias, minimizando a interrupção dos usuários. Da perspectiva do servidor, uma atualização de canary é a mesma que uma atualização de rolagem padrão, exceto que o número de instâncias que devem ser atualizadas é menor que o tamanho total do grupo de instâncias. Como uma atualização padrão, uma atualização canária é prejudicial para as instâncias afetadas; ou seja, as instâncias afetadas são excluídas e substituídas por novas instâncias de VM durante a atualização.

Criptografia em Repouso
O Google Cloud Platform criptografa o conteúdo do cliente armazenado em repouso, sem nenhuma ação necessária do cliente, usando um ou mais mecanismos de criptografia. Existem algumas pequenas exceções, observadas mais adiante neste documento.
Os dados para armazenamento são divididos em partes e cada parte é criptografada com uma chave de criptografia de dados exclusiva. Essas chaves de criptografia de dados são armazenadas com os dados, criptografados com chaves de criptografia de chave ("envolvidas" por) que são exclusivamente armazenadas e usadas dentro do serviço central de gerenciamento de chaves do Google. O serviço de gerenciamento de chaves do Google é redundante e globalmente distribuído.
Os dados armazenados no Google Cloud Platform são criptografados no nível de armazenamento usando AES256 ou AES128.
O Google usa uma biblioteca criptográfica comum, a Tink, para implementar a criptografia de forma consistente em quase todos os produtos do Google Cloud Platform. Como essa biblioteca comum é amplamente acessível, apenas uma pequena equipe de criptógrafos precisa implementar e manter adequadamente esse código rigorosamente controlado e revisado.
Criptografia em Trânsito
O Google criptografa e autentica todos os dados em trânsito em uma ou mais camadas da rede quando os dados se movem fora dos limites físicos não controlados pelo Google ou em nome do Google. Os dados em trânsito dentro de um limite físico controlado por ou em nome do Google geralmente são autenticados, mas não necessariamente criptografados.
Dependendo da conexão que está sendo feita, o Google aplica proteções padrão aos dados em trânsito. Por exemplo, protegemos as comunicações entre o usuário e o Google Front End (GFE) usando o TLS.
Os clientes do Google Cloud com requisitos adicionais para a criptografia de dados pela WAN podem optar por implementar proteções adicionais para dados à medida que eles se movem de um usuário para um aplicativo ou de uma máquina virtual para uma máquina virtual. Essas proteções incluem túneis IPsec, Gmail S / MIME, certificados SSL gerenciados e Istio.
Chaves de Criptografia
O Cloud Storage sempre criptografa seus dados no servidor, antes de serem gravados no disco, sem custo adicional. Além desse comportamento padrão, há outras maneiras de criptografar seus dados ao usar o Cloud Storage:
Chaves de criptografia fornecidas pelo cliente: você pode criar e gerenciar suas próprias chaves de criptografia para criptografia do lado do servidor, que atuam como uma camada adicional de criptografia sobre a criptografia padrão do Cloud Storage.
Chaves de criptografia gerenciadas pelo cliente: você pode gerar e gerenciar suas chaves de criptografia usando o Cloud Key Management Service, que atua como uma camada adicional de criptografia sobre a criptografia padrão do Cloud Storage. Suas chaves de criptografia são armazenadas no Cloud KMS. O projeto que contém suas chaves de criptografia pode, então, ser independente do projeto que contém seus blocos, permitindo assim uma melhor separação de tarefas.
Quando você aplica uma chave de criptografia gerenciada pelo cliente a um objeto, o Cloud Storage usa a chave ao criptografar:

Os dados do objeto.
A soma de verificação CRC32C do objeto.
O hash do MD5 do objeto.
O Cloud Storage usa chaves padrão do servidor para criptografar os metadados restantes do objeto, incluindo o nome do objeto. Isso permite que você leia e atualize metadados gerais, além de listar e excluir objetos, sem precisar da chave de criptografia gerenciada pelo cliente. No entanto, para executar qualquer uma dessas ações, você deve ter permissão suficiente para isso.
Criptografia do lado do cliente: criptografia que ocorre antes de os dados serem enviados para o Cloud Storage. Esses dados chegam ao Cloud Storage já criptografados, mas também sofrem criptografia no servidor.

Internet Key Exchange (IKE)
O Internet Key Exchange (IKE, às vezes, IKEv1 ou IKEv2, dependendo da versão) é o protocolo usado para configurar uma associação de segurança (SA) no conjunto IPsecprotocol. O IKE baseia-se no protocolo da Oakley e o ISAKMP.IKE usa certificados X.509 para autenticação - pré-compartilhados ou distribuídos usando DNS (preferencialmente com DNSSEC) - e uma troca de chaves Diffie-Hellman para configurar um segredo de sessão compartilhado a partir do qual chaves criptográficas são derivados.

Em outras palavras, as técnicas para troca de Chaves no IPSEC ... lembre-se apenas que elas precisam de um segredo de sessão compartilhada 
VPN IPsec
Na computação, o IPsec (Internet Protocol Security) é um conjunto de protocolos de rede seguro que autentica e criptografa os pacotes de dados enviados por uma rede IPv4. O pacote IPv4 inicial foi desenvolvido com tão poucas provisões de segurança que a versão IP estava incompleta, aberta ou deixada para desenvolvimento de pesquisa adicional. O IPsec inclui protocolos para estabelecer autenticação mútua entre agentes no início de uma sessão e negociação de chaves criptográficas para uso durante a sessão. O IPsec pode proteger os fluxos de dados entre um par de hosts (host para host), entre um par de gateways de segurança (de rede para rede) ou entre um gateway de segurança e um host (de rede para host).
Um túnel VPN conecta dois gateways VPN e serve como um meio virtual através do qual o tráfego criptografado é passado. Dois túneis VPN devem ser estabelecidos para criar uma conexão entre dois gateways VPN: Cada túnel define a conexão a partir da perspectiva de seu gateway, e o tráfego só pode passar quando o par de túneis é estabelecido. Um túnel de VPN na nuvem está sempre associado a um recurso de gateway da VPN de nuvem específico.

Migração ao vivo
O Compute Engine oferece migração ao vivo para manter as instâncias da máquina virtual em execução mesmo quando ocorre um evento do sistema host, como uma atualização de software ou hardware. O Compute Engine Live migra suas instâncias em execução para outro host na mesma zona, em vez de exigir que suas VMs sejam reinicializadas. Isso permite que o Google realize a manutenção que é essencial para manter a infraestrutura protegida e confiável sem interromper nenhuma de suas VMs.
A migração ao vivo mantém suas instâncias em execução durante:
Manutenção de infra-estrutura regular e upgrades.
Manutenção de redes e redes elétricas nos centros de dados.
Hardware com falha, como memória, CPU, placas de interface de rede, discos, energia e assim por diante. Isso é feito com base no melhor esforço; se um hardware falhar completamente ou impedir a migração ao vivo, a VM trava e reinicia automaticamente e um erro de host é registrado.
Atualizações do sistema operacional e do BIOS do host.
Atualizações relacionadas à segurança, com a necessidade de responder rapidamente.
Alterações na configuração do sistema, incluindo a alteração do tamanho da partição raiz do host, para armazenamento da imagem do host e dos pacotes.
A migração ao vivo não altera nenhum atributo ou propriedade da própria VM. O processo de migração ao vivo apenas transfere uma VM em execução de uma máquina host para outra máquina host na mesma zona. Todas as propriedades e atributos da VM permanecem inalterados, incluindo endereços IP internos e externos, metadados de instância, dados e volumes de armazenamento em block, estado do SO e do aplicativo, configurações de rede, conexões de rede e assim por diante.

Http (s) de balanceamento de carga
 
Um balanceador de carga HTTP completo é estruturado da seguinte maneira:
Uma regra de encaminhamento global direciona solicitações recebidas para um proxy HTTP de destino.
O proxy HTTP de destino verifica cada solicitação em um mapa de URL para determinar o serviço de back-end apropriado para a solicitação.
O serviço de back-end direciona cada solicitação para um back-end apropriado com base na capacidade de serviço, na zona e na integridade da instância de seus back-ends conectados. A integridade de cada instância de back-end é verificada usando uma verificação de integridade de HTTP, uma verificação de integridade de HTTPS ou uma verificação de integridade de HTTP / 2. Se o serviço de back-end estiver configurado para usar uma verificação de integridade HTTPS ou HTTP / 2, a solicitação será criptografada a caminho da instância de back-end.
Sessões entre o balanceador de carga e a instância podem usar o protocolo HTTP, HTTPS ou HTTP / 2. Se você usar HTTPS ou HTTP / 2, cada instância nos serviços de back-end deverá ter um certificado SSL.
HTTPS
Um balanceador de carga HTTPS usa um proxy HTTPS de destino em vez de um proxy HTTP de destino.
Um balanceador de carga HTTPS requer pelo menos um certificado SSL assinado instalado no proxy HTTPS de destino para o balanceador de carga.
A sessão SSL do cliente é finalizada no balanceador de carga.
Os balanceadores de carga HTTPS suportam o protocolo da camada de transporte QUIC.

Paxos
Paxos é uma família de protocolos para resolver consenso em uma rede de processadores não confiáveis. Consenso é o processo de concordar com um resultado entre um grupo de participantes. Esse problema se torna difícil quando os participantes ou seu meio de comunicação podem experimentar falhas.
A família de protocolos Paxos inclui um espectro de compensações entre o número de processadores, o número de atrasos de mensagens antes de aprender o valor acordado, o nível de atividade de participantes individuais, o número de mensagens enviadas e os tipos de falhas.

O balanceamento de carga HTTPS oferece suporte ao protocolo QUIC em conexões entre o balanceador de carga e os clientes. O QUIC é um protocolo de camada de transporte que fornece controle de congestionamento semelhante ao TCP e segurança equivalente a SSL / TLS para HTTP / 2, com desempenho aprimorado. Console Serial
Ative o acesso interativo ao console serial de uma instância para depurar problemas de inicialização e de rede, solucionar problemas de instâncias com problemas, interagir com o GRUB Unified Bootloader (GRUB) e executar outras tarefas de solução de problemas.
Uma instância de máquina virtual possui quatro portas seriais virtuais. Interagir com uma porta serial é semelhante ao uso de uma janela de terminal, em que a entrada e saída é inteiramente em modo de texto e não há interface gráfica ou suporte ao mouse. O sistema operacional da instância, o BIOS e outras entidades no nível do sistema geralmente gravam a saída nas portas seriais e podem aceitar entradas como comandos ou respostas a prompts. Normalmente, essas entidades no nível do sistema usam a primeira porta serial (porta 1) e a porta serial 1 é geralmente chamada de console serial.
Por padrão, você pode chamar o método getSerialPortOutput para ler as informações que sua instância gravou em suas portas seriais, mas não pode gravar informações para sua instância ler. No entanto, se você tiver problemas ao acessar sua instância por SSH ou precisar solucionar problemas de uma instância que não esteja totalmente inicializada, poderá ativar o acesso interativo ao console serial, que permite conectar e interagir com qualquer uma das portas seriais da sua instância. Por exemplo, você pode executar comandos diretamente e responder a prompts na porta serial.

Uma conta de serviço é uma identidade que uma instância ou um aplicativo pode usar para executar solicitações de API em seu nome. Essa identidade é usada para identificar aplicativos em execução nas suas instâncias de máquina virtual para outros serviços do Google Cloud Platform. Por exemplo, se você escrever um aplicativo que leia e grave arquivos no Google Cloud Storage, ele deve primeiro se autenticar na API do Google Cloud Storage. Em seguida, você atualizaria o código do seu aplicativo para passar as credenciais da conta de serviço para a Cloud Storage API. Seu aplicativo autentica sem problemas na API sem incorporar nenhuma chave secreta ou credencial de usuário em sua instância, imagem ou código do aplicativo.
Você pode usar contas de serviço para criar instâncias e outros recursos. Se você criar um recurso usando uma conta de serviço, esse recurso será propriedade da conta de serviço de criação. Você também pode alterar a conta de serviço de uma instância existente.

O Serviço de Transferência de Armazenamento transfere dados de uma fonte de dados on-line para um coletor de dados. Sua fonte de dados pode ser um bucket do Amazon Simple Storage Service (Amazon S3), um local HTTP / HTTPS ou um intervalo do Cloud Storage. Seu data sink (o destino) é sempre um intervalo do Cloud Storage.

Faça backup dos dados em um intervalo do Cloud Storage de outros provedores de armazenamento.

O Serviço de Transferência de Armazenamento realiza uma transferência de dados com uma operação de transferência. As operações de transferência são agendadas e configuradas por meio de um trabalho de transferência. O Serviço de Transferência de Armazenamento possui opções que facilitam as transferências de dados e a sincronização entre fontes de dados e depósitos de dados. Por exemplo, você pode:
Programar operações de transferência única ou operações de transferência recorrentes.
Exclua objetos existentes no intervalo de destino se eles não tiverem um objeto correspondente na origem.
Excluir objetos de origem depois de transferi-los.

Por padrão, o Serviço de Transferência de Armazenamento copia um arquivo da fonte de dados se o arquivo não existir no coletor de dados ou se for diferente entre a versão na origem e o coletor. O padrão também é reter arquivos na origem após a operação de transferência.

Migração de Tráfego / Divisão
Gerencie quanto tráfego é recebido por uma versão de seu aplicativo migrando ou dividindo o tráfego.
A migração de tráfego alterna suavemente o roteamento de solicitação, movendo gradualmente o tráfego das versões que atualmente recebem tráfego para uma ou mais versões especificadas.
A divisão de tráfego distribui uma porcentagem do tráfego para versões de seu aplicativo. Você pode dividir o tráfego para mover 100% do tráfego para uma única versão ou para rotear porcentagens de tráfego para várias versões. A divisão do tráfego para duas ou mais versões permite que você conduza testes A / B entre suas versões e fornece controle sobre o ritmo ao liberar recursos.
Lembre-se: a divisão de tráfego é aplicada a URLs que não segmentam explicitamente uma versão. Por exemplo, as URLs a seguir dividem o tráfego porque segmentam todas as versões disponíveis no serviço especificado:
[MY_PROJECT_ID] .appspot.com - Distribui o tráfego para versões do serviço padrão.
[MY_SERVICE]. [MY_PROJECT_ID] .appspot.com - distribui o tráfego para versões do serviço MY_SERVICE.
Para mais informações, consulte Como as solicitações são roteadas.
Para realizar manualmente a migração de tráfego e a divisão do console do GCP, consulte Migração de tráfego e divisão de tráfego.

Máquina Virtual / Instâncias
Uma instância é uma máquina virtual (VM) hospedada na infraestrutura do Google. Você pode criar uma instância usando o Console do Google Cloud Platform ou a ferramenta de linha de comando gcloud.
As instâncias do Compute Engine podem executar as imagens públicas para Linux e Windows Server fornecidas pelo Google, bem como imagens personalizadas personalizadas que você pode criar ou importar de seus sistemas existentes. Você também pode implantar contêineres do Docker, que são iniciados automaticamente em instâncias que executam a imagem pública do OS do Container-Optimized.


Você pode usar um conjunto de tipos de máquinas predefinidos ou criar seus próprios tipos de máquina personalizados.
Nuvem privada virtual
Uma rede VPC, às vezes chamada apenas de "rede", é uma versão virtual de uma rede física, como uma rede de data center. Ele fornece conectividade para suas instâncias de máquina virtual (VM) do Compute Engine, clusters do Kubernetes Engine, instâncias do App Engine Flex e outros recursos em seu projeto.
Fornece uma topologia de rede privada com políticas de alocação de IP, roteamento e firewall de rede para criar um ambiente seguro para suas implantações.
VPC compartilhado 
A VPC compartilhada permite que uma organização conecte recursos de vários projetos a uma rede VPC comum, para que eles possam se comunicar de forma segura e eficiente usando IPs internos dessa rede.
A VPC compartilhada permite que os administradores da organização deleguem responsabilidades administrativas, como a criação e o gerenciamento de instâncias, para os Administradores de Projetos de Serviços, enquanto mantêm o controle centralizado dos recursos de rede, como sub-redes, rotas e firewalls. 
VPN, Cloud VPN
O Cloud VPN conecta sua rede local com segurança à sua rede virtual privada (VPC) do Google Cloud Platform (GCP) por meio de uma conexão VPN IPsec. O tráfego viajando entre as duas redes é criptografado por um gateway de VPN e, em seguida, descriptografado pelo outro gateway de VPN. Isso protege seus dados enquanto eles viajam pela Internet.


Para criar uma rede privada virtual (VPN), consulte Escolhendo uma opção de roteamento VPN.
Gateway de VPN
Cloud VPN gateway
Um gateway de VPN virtual em execução no GCP gerenciado pelo Google, usando uma configuração especificada em seu projeto. Cada gateway do Cloud VPN é um recurso regional que usa um endereço IP externo regional. Um gateway do Cloud VPN pode se conectar a um gateway de VPN local ou outro gateway do Cloud VPN.

Gateway de VPN no local
O gateway VPN que não está no GCP, conectado a um gateway do Cloud VPN, pode ser um dispositivo físico em seu data center ou uma oferta de VPN física ou baseada em software na rede de outro provedor de nuvem. As instruções do Cloud VPN são escritas do ponto de vista da sua rede VPC, portanto, o "gateway local" é o gateway que se conecta ao Cloud VPN.

Túnel VPN
Um túnel VPN conecta dois gateways VPN e serve como um meio virtual através do qual o tráfego criptografado é passado. Dois túneis VPN devem ser estabelecidos para criar uma conexão entre dois gateways VPN: Cada túnel define a conexão a partir da perspectiva de seu gateway, e o tráfego só pode passar quando o par de túneis é estabelecido.

Visão
O Google Cloud Vision permite que os desenvolvedores entendam o conteúdo de uma imagem encapsulando modelos poderosos de aprendizado de máquina em uma API fácil de usar. Ele rapidamente classifica as imagens em milhares de categorias (por exemplo, "veleiro", "leão", "Torre Eiffel"), detecta objetos individuais e faces dentro de imagens, e encontra e lê palavras impressas contidas nas imagens. Você pode criar metadados em seu catálogo de imagens, moderar conteúdo ofensivo ou ativar novos cenários de marketing por meio da análise de sentimento de imagem. Você também pode analisar as imagens enviadas na solicitação e integrá-las ao seu armazenamento de imagens no Google Cloud Storage.

Fio - ainda outro negociador de recursos
A idéia fundamental do YARN é dividir as funcionalidades de gerenciamento de recursos e agendamento / monitoramento de tarefas em daemons separados. A idéia é ter um ResourceManager global (RM) e ApplicationMaster (AM) por aplicativo. Um aplicativo é um único trabalho ou um DAG de jobs.

O ResourceManager e o NodeManager formam a estrutura de computação de dados. O ResourceManager é a autoridade final que arbitra recursos entre todos os aplicativos no sistema. O NodeManager é o agente de estrutura por máquina responsável pelos contêineres, monitorando seu uso de recursos (cpu, memória, disco, rede) e relatando o mesmo para o ResourceManager / Scheduler.

Wireshark 
O Wireshark é o principal analisador de protocolos de rede do mundo. Ele permite que você veja o que está acontecendo em sua rede em um nível microscópico. É o padrão de fato (e frequentemente de jure) em muitos setores e instituições educacionais.

O Wireshark é um analisador de pacotes gratuito e de código aberto. Ele é usado para solução de problemas, análise, desenvolvimento de protocolos de software e comunicações e educação. Originalmente chamado Ethereal, o projeto foi renomeado Wireshark em maio de 2006 devido a problemas de marca registrada.

O Wireshark on Cloud é executado no Amazon Web Services (AWS) e o Google Cloud Platform (GCP) é usado para solução de problemas de rede, análise, desenvolvimento e educação de protocolos de software e comunicações.

O Wireshark é de propriedade da Wireshark (https://www.wireshark.org/) e possui todas as marcas registradas e direitos de PI relacionados a este software.

Windows no Compute Engine
Você pode executar seus aplicativos do Windows no Google Compute Engine e aproveitar os muitos benefícios disponíveis para instâncias de máquinas virtuais, como opções de armazenamento confiáveis, a velocidade da rede do Google e o escalonamento automático.
O Compute Engine fornece várias ferramentas para ajudar a trazer seus aplicativos e serviços do Windows para a nuvem:
Use as imagens do Windows Server para criar instâncias com um ambiente básico do Windows, sobre o qual você pode criar seus aplicativos. Para imagens do Windows Server 2016 e 2012 R2, você pode selecionar as configurações do Windows Server com Desktop Experience ou do Windows Server Core.
Use imagens do SQL Server para iniciar instâncias com o Windows Server com o SQL Server pré-instalado. Pague pelas licenças do Windows Server e do SQL Server somente quando você as usar. As imagens do Windows Server recebem faturamento por segundo e as imagens do SQL Server recebem faturamento por minuto.
Execute aplicativos .NET em suas instâncias do Compute Engine.
Implemente o Active Directory em suas instâncias e leve seus serviços de domínio para a nuvem.
Execute servidores da Web IIS para hospedar seu conteúdo da Web em instâncias do Windows.
Se você tiver licenças existentes para o SQL Server ou outros aplicativos executados em um ambiente Windows, use as licenças de aplicativos Microsoft existentes por meio do programa Microsoft License Mobility.
